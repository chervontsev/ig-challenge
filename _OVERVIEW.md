

### Папки

`/app` - Содержит корневой index приложения, роутер, который побключает компоненты из `domains`, глобальный стор (с токеном).
`/data` - Содержит сорсы по API, кэширование, сервисы с нормализацией. Потенциально здесь могут лежать словари.
`/domains` – То что имеет бизнес-контекст: Композиционные компоненты, сторы, процессинг, которые обусловлены бизнес требованиями.
`/general` – То что не имеет бизнес-контекста: UI компоненты, хуки общего пользования, утилиты.



### Сущности

#### Data слой

`.source.ts` - Источник данных. В данном случае коммуникация с сервером через API.
`.schema.ts` – Валидация схем. В данном демо используется Valibot, который, помимо валидации, также маппит DTO в Доменные стабильные данные.

#### Domain слой

`.controller.ts` – Оркестрирует процессы между сущностями.
`.store.ts` – Модульные сторы на Zustand.
`.feature.tsx` – Композицонные компоненты, обусловленные бизнес-контекстом.
`.page.tsx` – Композицонные компоненты, обусловленные бизнес-контекстом. Привязываются к route-ам.

#### Сущности общего характера

`.service.ts` - Коллекции функций, которые выполняют процессинг данных.
`.model.ts` – Типы, константы, статичные данные



### Маршруты процессов

#### Domain-Server
- Domain `feature` или `page` обращается к Domain `controller`.
- Domain `controller` обращается к Data `service`.
- Data `service` обращается к Data `source`.
- Data `service` валидирует и нормализует данные для Domain.
- Domain `controller` получает стабильные данные и передает их в Domain `store`.
- Domain `feature` или `page` подписаны на изменения в Domain `store` и получают данные.

#### Domain-State

Если нет необходимости в процессинге, `feature` и `page` могуг общаться со своими `store` напрямую.



### Общие свойства

`/domains` представляет собой вертикальные слайсы – то что обусловлено бизнес-контекстом живет рядом.
Это создает неравномерную структуру. Но в то же время это отражает реальный объем и комплексность 
каждого бизнес-контекста – контексты могут быть неравномерны естественным образом, страницы авторизации
и продуктовая страница имеют разные особенности.

Этот подход имеет хорошие свойства для масштабирования. Каждый домен может переехать в свой артефакт.
Data слой может стать самостоятельным артефактом. Все что лежит в `/general` может стать самостоятельными библиотеками.

Сущности на модульном уровне позволяют тестировать маршруты процессов без затрагивания UI.
